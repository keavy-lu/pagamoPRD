### PM Wireframe Checklist

| 項目                                              | 狀態 |
| ----------------------------------------------- | -- |
| 機制、介面的重點與目的要清楚傳達給設計                             |    |
| 資訊架構是否有遺漏，是否完整呈現所有資訊                            |    |
| 介面資訊是否分類有序、每個區塊呈現對應的資訊和功能                       |    |
| 文字或示意圖是否足夠傳達出介面如何操作                             |    |
| 文字或示意圖是否足夠傳達出介面如何與使用者互動                         |    |
| 是否清楚說明每個按鈕、欄位...的上下步驟與流程                        |    |
| 所有介面按鈕、欄位...會呈現的狀態有哪些                           |    |
| 中途跳出/關閉會怎麼處理                                    |    |
| 正確操作流程順一遍                                       |    |
| 錯誤操作流程有哪些，操作後的錯誤訊息有哪些                           |    |
| 若介面有非官方人員使用需提供翻譯文字，讓設計注意介面英翻不能爆版                |    |
| 不能被改變、必須保留的設計重點要告知設計                            |    |
| Wireframe製作時自己有疑慮的部分，可特別說明，請設計協助想出更好的UX/UI呈現方式  |    |
| 確認 wireframe 運營有共識，可壓時間請運營回覆                    |    |



### Design Draft Checklist&#x20;

|                                                                                              |   |
| -------------------------------------------------------------------------------------------- | - |
| 確認PM功能描述的完整性：文字、 UI Flow、功能邏輯、頁面狀態等                                                          |   |
| 確認已製作的頁面中的 GUI/UI Flow/UX 等是否與 PM 需求一致                                                       |   |
| 介面功能操作是否符合PM需求邏輯                                                                             |   |
| 確認所有 comment 都有被追蹤（解決後關閉）                                                                    |   |
| 是否包含所有 Mobile 和 App 流程、畫面差異                                                                  |   |
| 視覺風格是否符合遊戲風格，元件操作是否與其他介面一致                                                                   |   |
| 後台介面是否與其他後台類似排版、操作有一致性                                                                       |   |
| 版面上的標題、文字、圖片配置、尺寸、按鈕、間距...是否適配                                                               |   |
| 介面的重點是否有被突顯，主次是否分明                                                                           |   |
| 介面是否能讓使用者正確操作互動，是否有錯誤引導                                                                      |   |
| 設計若調整了文字，進行討論確認是否提交第二版送翻版本                                                                   |   |
| 設計若大幅調整介面配置，進行討論確認修改或維持                                                                      |   |
| <p>設計若做出和預排會議結論邏輯不同的 draft 設計，PM隨時在群組和RD確認是否可以修改調整 <br>附上介面說明文字/介面狀態在Draft，方便設計檢視畫面有無缺漏 </p> |   |
| PM/設計 draft 版本有共識後（可壓時間），群組請RD確認 draft 是否OK，RD確認後，才製作 Handoff                                |   |
